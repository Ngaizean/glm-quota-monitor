## 总览

将项目拆分为 **6 个阶段**，每阶段产出可验证的交付物。阶段间有依赖关系，但同一阶段内的任务可以并行。

```
Phase 0 ──→ Phase 1 ──→ Phase 2 ──→ Phase 3 ──→ Phase 4 ──→ Phase 5
脚手架       Rust 后端     macOS 集成     前端 UI       业务逻辑       打包发布
```

## Phase 0：项目脚手架

**目标**：跑通 Tauri 2.0 + React 空项目，macOS 菜单栏模式可用。

| # | 任务 | 交付物 | 验证标准 |

| 0.4 | 搭建前端脚手架（React + TypeScript + Tailwind） | 前端编译通过、热更新可用 | 修改 TSX 文件后自动刷新 |

**阶段交付**：一个无 Dock 图标、有菜单栏图标的空壳应用。

## Phase 1：Rust 后端核心

**目标**：API 调用通、数据库建表、IPC 通道可用。

| # | 任务 | 交付物 | 验证标准 |
|---|------|--------|----------|
| 1.1 | 定义 API 数据类型（`api/types.rs`） | QuotaLimit / ModelUsage / ToolUsage 结构体 + 反序列化 | 单元测试通过 |
| 1.2 | 实现 API 客户端（`api/client.rs`） | 三个接口的 HTTP 调用，Bearer Token 认证 | `cargo test` 用真实 API Key 调通返回 200 |
| 1.3 | 实现数据库模块（`db/`） | SQLite 建表迁移（accounts / usage_snapshots / alert_rules / alert_history）+ CRUD | 首次启动自动建表，增删改查测试通过 |
| 1.4 | 实现加密模块（`crypto.rs`） | AES-256-GCM 加密/解密 API Key，密钥存 macOS Keychain | 加密后解密还原正确，Keychain 读写正常 |
| 1.5 | 定义 IPC 命令（`commands/`） | account / quota / history 三组 Tauri Command | 前端能通过 `invoke()` 调用并拿到返回值 |

**阶段交付**：后端全部逻辑可独立测试，API 数据能拿到并写入数据库。

## 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| Tauri 2.0 Popover 支持不完善 | 状态栏弹出体验受限 | 备选：用独立无边框窗口模拟 Popover |
| macOS Keychain API 在 Rust 中不稳定 | 加密存储方案需调整 | 备选：使用 macOS Security.framework 通过 FFI |
| 智谱 API 变更或限流 | 数据获取失败 | 增加重试机制和本地缓存降级 |
| Apple 开发者账号未就绪 | 5.6 代码签名无法完成 | 先发未签名版，签名后补 |

---

## 技术决策（已确认）

| 决策项 | 方案 | 理由 |
|--------|------|------|
| Popover 实现 | 无边框窗口 + 手动定位到图标下方 | Tauri 无原生 Popover，业界标准做法 |
| 图表库 | Recharts | 体积小、API 简洁，迷你折线图够用 |
| Keychain 集成 | `keychain-rs` crate | 跨平台一行代码搞定，Windows 自动切 Credential Manager |
| 代码签名 | Phase 5 再定，先发未签名版 | 开发阶段不影响，后续根据是否有 Apple Developer 账号决定 |
